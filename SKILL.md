## 🚨 1. กฎเหล็กทางเทคนิค (Strictly Prohibited)

- **ห้ามใช้ Vanilla JS**: ต้องใช้ React State/Ref/Hooks เท่านั้น (เว้นแต่เป็นส่วนรับ/รันโค้ด อนุญาตให้ใช้ Plugin หรือแพ็กเกจที่คนอื่นสร้างไว้มาใช้แทนได้)
- **ห้ามใช้ระบบเสียงหรือเครื่องสังเคราะห์เสียง**: ห้ามใส่ฟังก์ชัน `playSound`, คลื่นเสียง Web Audio API, หรือปุ่มควบคุมเสียงเอฟเฟกต์ใดๆ ทั้งสิ้นในทุกไฟล์คอมโพเนนต์ เพื่อความเงียบและเสถียรในการทำงานแบบออฟไลน์
- **ห้ามใช้ Tailwind แบบ Dynamic**: ให้ Hardcode string เต็มรูปแบบ (เช่น `text-indigo-600` ห้ามใช้ `${color}-600`) เพื่อรองรับ Tailwind v4
- **ห้ามแตะ Core Architecture**: ห้ามแก้ไข `\Teach\LMS-React\src\App.jsx`, `\Teach\LMS-React\src\components\LessonViewer.jsx`, `\Teach\LMS-React\vite.config.js` เว้นแต่จะมีระบุในคำสั่ง
- **ห้าม Card-in-Card**: ทุกหน้าต้องเป็นแบบไร้ขอบ (Seamless) ผสานไปกับฉากหลังตาม Immersive Full-Page Standard
- **State Management**: ห้ามใช้ Global State สำหรับ Simulator เด็ดขาด ให้ใช้ Local State (`useState`, `useEffect`) เสมอเพื่อความเป็นอิสระ

## 💻 2. มาตรฐานการเขียนโค้ด (Coding Standards)

- **Component Format**:
  - ให้เขียนเป็น `export default function ComponentName() { ... }`
  - การตั้งชื่อไฟล์: อ้างอิงตามวิชาและหมายเลขบทเรียนลำดับที่ 1 จากหลักสูตร (Curriculum) เสมอ โดยการนำคำย่อของรายวิชา (ตัวพิมพ์เล็ก) มาเชื่อมกับหมายเลขหลักสูตรด้วยเครื่องหมายขีดล่าง `_` และไม่มีคำอธิบายเพิ่มเติม ตัวอย่างเช่น:
    - **วิชา Python (21902-2001)**: หัวข้อ 1.1 ตั้งเป็น `py1_1.jsx`, หัวข้อ 3.5 ตั้งเป็น `py3_5.jsx`
    - **วิชา SQL (21901-2001)**: หัวข้อ 1.2 ตั้งเป็น `sql1_2.jsx`, หัวข้อ 4.8 ตั้งเป็น `sql4_8.jsx`
    - **วิชา OOP (Python OOP)**: หัวข้อ 2.3 ตั้งเป็น `oop2_3.jsx`
    - **วิชา OOAD**: หัวข้อ 3.1 ตั้งเป็น `ooad3_1.jsx`
    - **วิชา DSA**: หัวข้อ 1.1 ตั้งเป็น `dsa1_1.jsx`
- **Interactive Component**: ทุกสื่อ Simulator ให้สร้างเก็บไว้ที่ `\Teach\LMS-React\src\components\interactive\<วิชา>\` (ระบบจะโหลดและนำไปใช้งานอัตโนมัติผ่าน Auto-Registry)
- **ภาษาและเนื้อหา**: โค้ดตัวอย่างในวิชาต้องใช้ **Python** หรือ **MySQL** และคำอธิบายเนื้อหาต้องเป็น **ภาษาไทย** เสมอ
- **JSX Gotchas**: หากมีสัญลักษณ์ `>>>` ในโค้ด ให้ห่อด้วย `{">>>"}` เสมอ
- **เลย์เอาต์แยกการ์ดรายหัวข้อ (Separated Cards Layout)**: ในแต่ละบทเรียนย่อยที่มีหัวข้อย่อยบังคับ (เช่น 4.1.1, 4.1.2, 4.1.3) **ห้าม** ยุบรวมเนื้อหาลงในแท็บ (Tab Navigation) สับเปลี่ยนหน้า ให้ **แยกเนื้อหาออกเป็นบล็อกการ์ดเดี่ยวอิสระเรียงแนวตั้ง (Vertical Stack)** เพื่อให้นักเรียนสามารถเลื่อนอ่านเนื้อหาทั้งหมดได้อย่างลื่นไหล และมีกิจกรรมโต้ตอบที่ชัดเจนในแต่ละหัวข้อย่อยแยกกันอย่างเป็นระบบ

## 🎮 3. แนวทาง Game-Based Learning & Interaction

- **Visual Interactive**: เน้นการสร้าง "Interactive ที่อธิบายภาพและทำงานได้จริง" ให้เห็นชัดเจน ไม่จำเป็นต้องทำระบบสะสมคะแนนที่ซับซ้อน (หมายเหตุ: ถ้าเนื้อหาใน list ระดับที่ 2 ของโฟลเดอร์ `\Teach\LMS-React\docs\curriculum\` สำหรับวิชาต่างๆ เช่น 1.1.1, 1.1.3, 4.5.6, 1.3.8 ไหนสามารถสร้างได้ ให้พยายามสร้างให้ครอบคลุมทุกหัวข้อยิ่งดี)
- **Flowcharts**: ใช้ SVG บริสุทธิ์ แทน Unicode หรือ CSS ให้วาดสวยงามระดับเดียวกับ draw.io
- **Micro-Interactions**: ปุ่มและการ์ดที่โต้ตอบได้ต้องใส่ `active:scale-98 transition-all` เสมอ
- **Controls**: ปุ่ม Play, Pause, และ Reset (ไอคอน `RotateCcw`) ให้พิจารณาใส่ตามความเหมาะสมของสถานการณ์ ไม่จำเป็นต้องบังคับใส่ทุกครั้ง

## 📚 4. หลักสูตรและการจัดการข้อมูล

- **โครงสร้างเนื้อหา**: อ้างอิงโฟลเดอร์ `\Teach\LMS-React\docs\curriculum\`
  - Unit = บทหลัก
  - ลำดับที่ 1 (เช่น 1.1) = จำนวนหน้าเว็บ (Lessons)
  - ลำดับที่ 2 (เช่น 1.1.1) = หัวข้อบังคับที่ต้องมีในหน้านั้น (**กฎเหล็ก: ห้ามใส่ตัวเลขลำดับย่อยเด็ดขาด!** เช่น ในการสร้างหน้า 4.3 เมื่อมีการแยกการ์ดเนื้อหารายหัวข้อ 4.3.1 และ 4.3.2 **ห้ามพิมพ์ตัวเลข 4.3.1 หรือ 4.3.2 ลงในหน้าเว็บ/คอมโพเนนต์เด็ดขาด** ให้ใส่เฉพาะชื่อหัวข้อที่เป็นข้อความเปล่าๆ เท่านั้น เพื่อความสะอาด สวยงาม และตรงตามสไตล์ของระบบ)
- **โครงสร้างข้อมูลบทเรียน (Data Structure Standard)**: ทุกบทเรียนในฐานข้อมูล **ต้องมี** คีย์ข้อมูลให้ครบทั้ง 6 ฟิลด์ ได้แก่ `id`, `title`, `mainTitle`, `subTitle`, `description`, และ `content` วางไว้ในอ็อบเจกต์ `lesson` เสมอ ห้ามขาดแม้แต่ฟิลด์เดียว เพื่อใช้ส่งไปแสดงผลที่ `StandardHeader` ได้อย่างสมบูรณ์และเป็นมาตรฐานเดียวกัน
- **ขั้นตอนการสร้างหน้าใหม่ (New Page Creation Workflow)**:
  1. **สร้างไฟล์ Component**: สร้างไฟล์ใน `\Teach\LMS-React\src\components\interactive\<วิชา>\` โดยตั้งชื่อตามรูปแบบ `<prefix><X>_<Y>.jsx` เท่านั้น (เช่น `py1_1.jsx`, `sql3_5.jsx` อ้างอิงตามวิชาและหมายเลขบทเรียนลำดับที่ 1 จากหลักสูตร) และ **ห้าม** เติมคำอธิบายอื่นในชื่อไฟล์เด็ดขาด
  2. **โครงสร้าง Component**:
     - **ห้าม** ใส่ส่วนหัว (`<header>`, `h1`, `h2` ที่บอกชื่อเรื่อง) ลงในไฟล์ Component เด็ดขาด เพราะระบบมี `StandardHeader` จัดการให้อัตโนมัติอยู่แล้ว
     - ให้เน้นเขียนเฉพาะส่วนที่เป็น `<main>` หรือตัว Simulator / มินิเกม / เนื้อหา และจบด้วย `<TeacherTask title="..." taskText="..." />` ที่มีรายละเอียดโจทย์กำกับไว้เสมอ
  3. **เชื่อมโยงข้อมูล**: นำชื่อไฟล์ Component ไปใส่ในรูปแบบ `[Marker]` ในฟิลด์ `content` ของไฟล์ Database (เช่น `pyCourse.js` ในรูปแบบ `[py1_1]` หรือ `[py3_5]`) พร้อมตรวจสอบให้แน่ใจว่าฟิลด์อื่นๆ ทั้ง 5 ฟิลด์ระบุข้อมูลครบถ้วน
  4. **การเขียนทับทันที (Overwrite Policy - ห้ามเปิดอ่านไฟล์เก่าเด็ดขาด)**: เมื่อผู้ใช้ระบุหรือสั่งให้สร้างหน้าบทเรียนใดขึ้นมาใหม่ ให้ AI ดำเนินการคิดออกแบบและสร้างโครงสร้างคอมโพเนนต์ใหม่ 100% ขึ้นมาเองตั้งแต่ต้น แล้วเขียนทับ (Overwrite) ลงบนไฟล์เดิมได้เลยทันที โดย**ห้ามเรียกใช้เครื่องมือเปิดอ่านไฟล์เก่าตัวเดิมก่อนลงมือทำอย่างเด็ดขาด** (ไม่จำเป็นและห้ามพยายามรักษาโครงสร้างหรือโค้ดเดิมใดๆ ไว้ เพื่อประสิทธิภาพและป้องกันไม่ให้ดีไซน์เดิมมารบกวนรูปแบบใหม่ที่สวยงามกว่าเดิม)
  5. **มาตรฐานการแจกแจงแผนงานและโค้ดหัวข้อย่อยย่อยให้ครบถ้วน (Comprehensive Sub-topic Planning & Coding Standard - อัปเดต 30 May 2026)**: เมื่อได้รับคำสั่งหรือตรวจพบว่าบทเรียนในหลักสูตรมีหัวข้อย่อยระดับที่ 2 (เช่น 1.1.1 - 1.1.7 หรือ 2.5.1 - 2.5.4) อยู่เท่าใด AI **ต้อง** จัดทำแผนการดำเนินงาน (Implementation Plan) และเขียนโค้ด Component จริง โดยแจกแจงเนื้อหาเป็นบล็อกหรือการ์ดอินเตอร์แอคทีฟแยกกันให้ครบถ้วนทุกๆ หัวข้อย่อย ห้ามทิ้ง ข้าม ละเว้น ยุบรวม หรือย่อทฤษฎีเด็ดขาด เพื่อคงความลึกซึ้งและข้อมูลระดับ Massive Scale สูงสุด
  6. **มาตรฐานการลบงานและรวมศูนย์เช็คลิสต์ (Unified & Clean Task Checklist Standard - อัปเดต 30 May 2026)**: รายการงานเช็คลิสต์ที่ทำเสร็จแล้ว (Completed) **ต้อง** ถูกลบออกจาก `\Teach\task.md` ในเซสชันทันที เพื่อให้แผนงานกระชับและไม่ยาวสับสน และเช็คลิสต์ของระบบ (เช่นไฟล์ `task.md` ใน App Data/Brain) **ต้อง** ถูกบันทึกรวมศูนย์ลงใน `\Teach\task.md` ของโครงการโดยตรง ห้ามสร้างไฟล์แยกเด็ดขาด

## 🧠 5. Continuous Training (การอัปเดตนิสัยการเขียนโค้ด)

- **ปรับตัวตามผู้ใช้**: สังเกตและจดจำรูปแบบการเขียนโค้ดหรือตรรกะที่ผู้ใช้พึงพอใจ หากผู้ใช้สั่งให้หลีกเลี่ยงการเขียนแบบใด หรือให้แก้การทำงานซ้ำๆ ให้คุณถือเป็น "บรรทัดฐานใหม่ (New Standard)" และเขียนทับลงในหมวดหมู่นี้ได้ทันที เพื่อผลิตผลงานให้ตรงใจมากที่สุด
- **New UI Fix (27 May 2026)**: ทุกครั้งที่ใช้ `text-transparent bg-clip-text` กับข้อความภาษาอังกฤษ/ไทยขนาดใหญ่ **ต้องใส่คลาส `pb-2 leading-normal` เสมอ** เพื่อแก้บัก Tailwind ตัดหางตัวอักษร (Descender Clipping) ที่ทำให้หางตัว p, y, j ขาดหายไป
- **Compact Header Spacing Standard (29 May 2026)**: ปรับปรุงความกระชับของส่วนหัว (StandardHeader) ทุกหน้า โดยกำหนดให้ใช้ Padding `p-6 md:p-8` สำหรับหน้าปกติ และ `pt-8 pb-4 md:pt-10 md:pb-5` สำหรับหน้าไร้ขอบ (immersive), ปรับระยะห่าง `subTitle` เป็น `mb-3 pb-2 leading-normal`, ย่นระยะห่างของเส้นคั่น `div` เหลือ `my-3`, และปรับแต่งบล็อกคำอธิบายสังเขปเป็น `border-l-[3px] border-teal-500/80 pl-3.5 py-0.5` พร้อมปรับฟอนต์เป็น `text-[15px] md:text-base text-slate-500` เสมอ เพื่อความสะอาด สวยงาม และประหยัดพื้นที่สายตาของผู้เรียน
- **Massive Content Standard (28 May 2026)**: ในการสร้างหน้าบทเรียนที่มีหัวข้อย่อยระดับที่ 2 หลายหัวข้อ (เช่น 2.8.1, 2.8.2) AI ต้องสร้างเนื้อหา คำอธิบาย และระบบโต้ตอบ (Interactive) แยกให้ชัดเจนแบบ "จัดเต็มและลึกซึ้ง (Massive Scale)" โดยแต่ละหัวข้อย่อยควรมีความยาวเนื้อหาและโค้ดอย่างน้อย 400 บรรทัด (โค้ดในไฟล์รวมอาจทะลุ 1,000+ บรรทัดได้) ห้ามเขียนแบบย่อ หรือทำแบบขอไปทีเด็ดขาด
- **TeacherTask Props Standard (28 May 2026)**: ห้ามเรียกใช้งานคอมโพเนนต์ `<TeacherTask />` เปล่าๆ เด็ดขาด! จะต้องใส่ props ควบคู่ไปด้วยเสมอ คือ `title` (ชื่อภารกิจ) และ `taskText` (คำอธิบายโจทย์และขั้นตอนการทำงาน) เพื่อให้นักเรียนมีเป้าหมายในการเรียนรู้
- **ANSI Display Symbol Fix (28 May 2026)**: การวาดสัญลักษณ์แสดงผลทางหน้าจอ (Display) ในผังงานด้วย SVG ให้ยึดรูปทรงมาตรฐานสากล คือด้านขวาแหลม (Pointed) และด้านซ้ายโค้งเว้าเข้า (Concave arc) เช่น โค้ด `path d="M 25,5 Q 10,25 25,45 L 75,45 L 95,25 L 75,5 Z"` ห้ามใช้รูปทรงอื่นที่ผิดเพี้ยนเด็ดขาด
- **Dynamic CSS Variable Flow Standard (29 May 2026)**: สำหรับระบบจำลองสายการไหล (Flowchart lines) ให้ประยุกต์ใช้ SVG `stroke-dasharray` ร่วมกับการตั้งค่า Inline CSS variable (เช่น `--flow-speed`) เพื่อเชื่อมโยงความเร็วแอนิเมชันกับ State/Slider ของ React เพื่อลดการใช้ทรัพยากรการประมวลผล
- **Interactive Repair & Morph Standard (29 May 2026)**: ในระบบอธิบายบั๊ก/ข้อผิดพลาด (Fault diagnosis) ให้สร้างแบบจำลองโต้ตอบที่มีปุ่ม **"ซ่อมแซม"** ซึ่งจะอัปเดตสลับโครงสร้าง SVG และแก้ไขตัวอย่างชุดโค้ดโปรแกรมคู่ขนานไปสู่รูปแบบที่ถูกต้อง พร้อมสร้างแอนิเมชันแฟลชนำทางด้วยสีสันเรืองแสงเพื่อประสิทธิภาพทางการเรียนรู้ขั้นสูง
- **Massive Content Generation Rule (29 May 2026)**: เมื่อได้รับคำสั่งให้สร้างเนื้อหาสำหรับหัวข้อย่อย (Sub-topic) ที่มีหมายเลขชัดเจน (เช่น 2.8.1, 2.8.2, 2.8.3...) ให้สร้างเนื้อหาในระดับ "Detailed In-Depth Explanation" โดยมีองค์ประกอบดังนี้:
  1. **Content Overview**: แนะนำเนื้อหาโดยละเอียด
  2. **Syntax Deep Dive**: แยกวิเคราะห์ไวยากรณ์ (Syntax) ของคำสั่ง/โครงสร้างนั้นๆ โดยเฉพาะ (ถ้ามี)
  3. **Python Code Examples (Real)**: ชุดตัวอย่างโค้ด Python ที่สามารถคัดลอกไปรันได้จริงและใช้งานร่วมกันได้
  4. **Interactive Practice**: ระบบทดสอบโต้ตอบ (Quiz/Simulation) ที่ใช้ Logic อ้างอิงจากเนื้อหาและโค้ดด้านบนทันที เพื่อให้ผู้ใช้สามารถฝึกฝนสิ่งที่เพิ่งเรียนรู้ไปได้ทันที (Instant Reinforcement)
  5. **Summary**: สรุปเนื้อหา
- **Pure SVG Flowchart & Junction Standards (29 May 2026)**: มาตรฐานการเขียนผังงานเชิงโต้ตอบด้วย SVG ต้องยึดหลักเกณฑ์ดังนี้เพื่อความสวยงามพรีเมียม:
  1. เส้นสายการวิ่งเชื่อมต่อ (`<path>`) ทั้งหมดต้องใส่คุณสมบัติ `fill="none"` เสมอ เพื่อป้องกันบักรูปทรงปิดถมดำทึบ (Black-Filled Polygon Bug)
  2. เมื่อมีแนวกระแสหลายสายไหลมาบรรจบกัน (Merge Points) ต้องขจัดช่องตัดเว้าให้เชื่อมโยงกันอย่างเสถียร และวาดจุดเชื่อมตรรกะแบบวงกลม `<circle cx="..." cy="..." r="..." />` เสมอ โดยใช้สีน้ำเงินอินดิโกสำหรับสถานีทิศทางหลัก และสีเทาสเลตสำหรับสายคัดกรองเงื่อนไขย่อย
  3. กำหนดโครงสร้างลูกศรนำทิศใน `<defs>` และเรียกใช้งานผ่าน `markerEnd` โดยคัดเลือกรหัสสีลูกศรให้ตรงตามสายตรรกะแบบแยกชัดเจน (น้ำเงิน = สายหลัก, เขียว = จริง, แดง/ส้ม = เท็จ, เทา = สายไม่เปิดใช้งาน)
- **Absolute Sound Prohibition Standard (30 May 2026)**: บัญญัติมาตรฐานการห้ามใช้เสียงเอฟเฟกต์ ตลอดจนปุ่มสลับเสียงและไลบรารี Web Audio API ในทุกหน้าและทุกคอมโพเนนต์อย่างถาวร 100% เพื่อลดความซับซ้อน ป้องกันบักการโหลดทรัพยากรออฟไลน์ และรักษาความสะอาดลื่นไหลของหน้าเว็บ
- **Proportional Font Size & Readability Standard (30 May 2026)**: มาตรฐานสเกลขนาดตัวอักษรของแพลตฟอร์มเพื่อการอ่านที่สบายตาของนักเรียน:
  - ข้อความคำอธิบายเนื้อหาและทฤษฎีทั่วไป: บังคับใช้ `text-[16px]` หรือ `text-base` ร่วมกับ `leading-relaxed`
  - ส่วนคำอธิบายปลีกย่อย, ปุ่มคีย์บอร์ดตัวเลือก, ป้ายข้อความ: บังคับใช้ขนาด `text-sm` หรือ `text-[13px]` เป็นขั้นต่ำ ห้ามใช้ `text-xs` หรือ `text-[11px]` เด็ดขาดเพื่อสุขภาวะการอ่าน
  - กล่องโค้ดภาษาและหน้าจอ Virtual Terminal: บังคับใช้ขนาดตัวอักษรขั้นต่ำ `text-[13px]` หรือ `text-[13.5px]` เสมอ
- **Compact Card Spacing Standard (30 May 2026)**: กำหนดมาตรฐานระยะห่างระหว่างการ์ดใน Layer 3 (Flexible Subtopics) ให้กระชับและสมดุลสายตายิ่งขึ้น โดยบังคับใช้ `space-y-6 md:space-y-8` ในแท็ก `<main>` เสมอ ห้ามเว้นระยะห่างกว้างเกินไปจนเคว้ง
- **Unified Background & Dynamic Ambient Standard (30 May 2026)**: กำหนดมาตรฐานพื้นหลังหลักทั่วทั้งบทเรียนเป็นสีเดียวกันเพื่อความลื่นไหลสม่ำเสมอ (เช่นสีโทนสว่างคงที่ #FAFAFA) โดยใช้ Layer 1 (Ambient Backdrop) เรืองแสงฟุ้งขวางขอบเฟรม และทำการสุ่มโทนสี Gradients ตามธีมรายวิชาหรือตามระดับเนื้อหา/หัวข้อย่อยอย่างเป็นอิสระ เพื่อสร้างมิติสีสันสวยงามเฉพาะตัวในแต่ละหน้านั้นๆ
- **Clickable Hover Visual Standard (30 May 2026)**: มาตรฐานการบอกเบาะแสให้ผู้เรียนรู้ว่าปุ่มหรือการ์ดนั้นคลิกโต้ตอบได้ (Affordance Indicator) บังคับใช้คลาส `cursor-pointer` ร่วมกับเอฟเฟกต์โฮเวอร์ เช่น `hover:scale-[1.02]` หรือ `hover:-translate-y-1` และ `hover:shadow-md` หรือ `hover:shadow-lg` ในปุ่มและการ์ดอินเตอร์แอคทีฟทั้งหมดทุกหน้าเพื่อการใช้งานลื่นไหลและเข้าใจการทำงานทันที
- **Glassmorphism & Pulsing Interactive Icon Standard (30 May 2026)**: บัญญัติมาตรฐานดีไซน์ระดับพรีเมียมสำหรับการอัปเกรดบทเรียน โดยทุกคอมโพเนนต์ต้องใช้โครงสร้างการ์ดกระจกใสกึ่งโปร่งแสง `bg-white/60 backdrop-blur-xl border border-white/40 shadow-xl` ผสานกับจานรองไอคอนแบบ `group` ที่ตอบสนองโฮเวอร์ด้วยการขยับ/หมุนเบาๆ (เช่น `group-hover:rotate-12 group-hover:scale-110`) หรือกะพริบชีพจร (`group-hover:animate-pulse`) เพื่อยกระดับความสวยงามและเพิ่มความน่าสนใจในการใช้งาน
- **Clean Reusable Data Mapping Standard (30 May 2026)**: บัญญัติมาตรฐานการเขียนโค้ดเพื่อลดความซ้ำซ้อนของคำสั่ง (Deduplication) โดยกำหนดให้แปลงเนื้อหาที่เป็นบล็อกโครงสร้างย่อยที่มีความคล้ายกัน ให้มาอยู่ในรูปแบบของอาร์เรย์โครงสร้างข้อมูล (Array of Objects) แล้วเรนเดอร์ออกทางหน้าจออย่างไดนามิกผ่านการ `.map()` และการใช้ Local State ร่วมกัน แทนการเขียนโค้ด HTML/JSX ซ้ำๆ แบบ Hardcode ช่วยลดความยาวของไฟล์ลงมากกว่า 50% เพิ่มประสิทธิภาพการบำรุงรักษาโค้ด แต่ยังคงรายละเอียดทฤษฎีและระบบโต้ตอบย่อยครบถ้วนสมบูรณ์ 100%
- **OO-style Composition Standard — Shared Base Components (30 May 2026, อัปเดต 30 May 2026)**: บัญญัติมาตรฐานการสร้างและนำ "แม่พิมพ์ (Base Components)" กลับมาใช้ซ้ำข้ามบทเรียน โดยเก็บไว้ที่ `\Teach\LMS-React\src\components\interactive\shared\` และนำเข้าผ่าน Barrel Export ที่ `shared/index.js` มีหลักการดังนี้:
  - **SimulatorShell** = แม่พิมพ์การ์ดหลัก (light/dark mode) พร้อม Glassmorphism + Hover animations — ใช้แทนโครงสร้างการ์ดที่เขียนซ้ำทุกบทเรียน
  - **ConsoleScreen** = กล่องจำลอง Python Terminal/Console — ใช้แทนการเขียน dark code output panel ซ้ำทุกการ์ด
  - **OptionSelector** = ปุ่มตัวเลือก Grid รองรับ pill/card mode — ใช้แทน button grid ที่ซ้ำกันทุกบทเรียน
  - **QuizEngine** = เกมตอบคำถาม Multiple Choice ที่จัดการ state ครบ (level/win/fail/reset) ด้วยตัวเอง — ลด ~225 บรรทัด เหลือแค่ ~40 บรรทัดข้อมูล
  - **AmbientBackdrop** = Layer 1 Background blobs พร้อม Theme Presets (PY1–4, SQL1) — ใช้แทนการ copy-paste วงกลมเรืองแสง fixed inset-0 ทุกบทเรียน
  - **ConceptCard** = การ์ดแนวคิดพรีเมียม สีขาว มีวงกลม accent ต่างสีมุมขวาบน (12 accent presets) + code strip + result badge — ใช้สำหรับ operators, keywords, data types ทุกบทเรียน
  - **SectionBlock** = กรอบส่วน gradient ล้อมรอบกลุ่ม concept cards — ใช้แบ่งกลุ่มหัวข้อในบทเรียนที่มีหลาย concept ต่างประเภท
  - **LogicGateSimulator** = จำลองวงจรตรรกะ boolean แบบ visual flow (input → operator → output) รองรับ binary (and/or/==) และ unary (not) มี toggle-able inputs และ formula bar ด้านล่าง
  - **กฎเหล็กการใช้งาน**: เมื่อสร้างบทเรียนใหม่ให้ **ต้อง** import จาก `../shared` ก่อนเสมอ และเขียน custom component เพิ่มเฉพาะส่วนที่ต้องการตรรกะเฉพาะของบทเรียนนั้น ๆ เท่านั้น เพื่อประสิทธิภาพสูงสุด
- **Concept Card Visual System Standard (30 May 2026)**: ดูรายละเอียดและ Semantic Accent Color Palette ครบถ้วนได้ที่ `\Teach\DESIGN.md` ส่วนที่ 5 (§5.1–5.5) — ห้ามใช้สีตามอำเภอใจ ต้องใช้สีตาม semantic ที่กำหนดเท่านั้น (เช่น == → amber, != → rose, and → purple, True → emerald)
